# adversarial-review skill 设计文档

## 背景与目标

现有的审查类 skill（`code-review`、`security-review`）都是针对**代码 diff**：找正确性问题、安全漏洞、可简化点。但审查**规格类文档**——OpenSpec change proposal、设计文档、PRD、RFC——需要一种不同的批判方式：这些文档描述的是"打算做什么、为什么这么做"，问题往往不在语法或实现细节，而在**假设是否站得住脚、方案是否经得起攻击、是否存在更好的替代方案、需求是否有歧义或遗漏**。

目标是做一个显式调用的 `adversarial-review` skill，用对抗性思维审查这类文档，产出结构化的发现列表。初期范围有意收窄在"规格/设计类文档"，验证审查维度和输出形式跑得通之后，再考虑扩展到更泛化的文本类型。

## 范围

**覆盖**：OpenSpec change proposal、设计文档、PRD、RFC 等"描述未来决策"的文档。

**不覆盖**（本次设计明确排除）：
- 代码 diff 的正确性/安全审查 —— 属于 `code-review` / `security-review` 的职责
- 纯叙述性文本（博客、邮件、会议纪要）—— 留待未来视需要再扩展
- 自动修改文档 —— 详见"只读边界"

## 命名与触发

- Skill 目录：`skills/adversarial-review/SKILL.md`
- 显式调用触发，不做关键词自动触发。理由：这是一个"重"操作（多阶段分析 + 代码交叉核实），用户随手起草文档时不应该被动触发一轮找茬；应该是用户明确决定"这份文档我想要被挑刺"时才启动。

## 输入处理

- 接受一个或多个文件/目录路径作为审查对象。
- 传入目录时（例如一个 OpenSpec change 目录），把目录下的文本类文件都当作同一个审查对象整体读入，不假设固定文件名或目录结构（不内建 OpenSpec 的 proposal/tasks/specs 结构知识）。
- 未显式指定路径时，向用户询问要审查哪个文档，不做猜测。

## 代码库上下文

如果目标文档所在的工作目录本身就是代码仓库的一部分，审查时主动使用 Grep/Read 核实文档中关于"现状"或"改动范围"的声称（例如"当前系统不支持 X"、"只需改动 Y 模块"），而不是只依赖文档自身的描述来判断真假。如果文档明显独立于任何代码库（纯文档仓库），跳过这一步。

## 审查流水线（五阶段）

1. **通读并列清单**：完整读完所有目标文档，提炼出一份待审事项清单——关键决策、隐含假设（技术/数据/组织前提）、关于现状的声称、验收标准与边界条件。这份清单是后续阶段的输入，避免漫无目的地通读。

2. **逐条套用四种对抗手法**：针对清单上每一项，选用合适的手法生成候选发现（一项可能触发多种手法）：
   - **假设挖掘**：这条假设如果不成立，方案会怎样？
   - **失败场景构造**：站在"想让方案失败"的攻击者角度，构造边界情况、误用方式、并发/时序问题、恶意输入、资源耗尽等场景。
   - **反方案 / 魔鬼代言人**：针对关键决策构造一个更有竞争力的替代方案，检验原方案的理由是否真的站得住脚。
   - **歧义 / 遗漏检测**：是否存在可以有两种合理理解的表述？验收标准是否可测？是否有边界条件完全未提及？

3. **代码交叉核实**（仅当能访问代码库时）：对涉及"现状声称"或"改动范围"的候选发现，用 Grep/Read 实际核实。

4. **自我验证过滤**：回头逐条检查阶段 2/3 产出的候选发现，判断是否真的站得住脚，标记 `CONFIRMED`（有明确依据）或 `PLAUSIBLE`（有道理但没有十足把握）。丢弃纯粹为了挑刺而挑刺、缺乏依据的项目。这是防止"对抗性"滑向"无脑找茬"的关键闸门。

5. **排序输出**：将剩余发现按严重度从高到低排列，通过 `ReportFindings` 工具输出。

不提供可调节的力度分级（不像 `code-review` 有 low/medium/high）。标准固定为："每条发现必须能追溯到文档里的具体文字，或与代码的具体矛盾，并通过阶段 4 的自我验证"——宁可少而准，不为凑数量硬找。

## 输出格式

复用 `ReportFindings` 工具，字段映射：

| 字段 | 含义 |
| --- | --- |
| `file` | 该发现来源的具体文档路径（多文件审查时，每条发现各自指向其实际出处，不能笼统指向目录） |
| `line` | 该发现对应的文档行号；找不到能精确定位的单行时（例如问题横跨整段或是"通篇未提及某边界条件"）可以省略，指向最相关段落的起始行即可，不必强行凑一个行号 |
| `summary` | 一句话说清问题是什么 |
| `failure_scenario` | 具体的失败场景/反例——对抗性审查的核心产出，必须具体，禁止"可能存在风险"式的空话 |
| `verdict` | `CONFIRMED`（有明确依据）/ `PLAUSIBLE`（有道理但没有十足把握，仍值得作者看一眼） |

## 只读边界

Skill 全程不编辑目标文档，只输出发现列表。SKILL.md 中会明确写出这条规则，防止未来被无意中扩展出 `--fix` 之类的自动修改行为——对抗性审查的价值在于保持独立、客观，"怎么改、改不改"应留给文档作者自己决定。

## 与现有 skill 的关系

- **`code-review` / `security-review`**：审查对象是代码 diff，找正确性/安全漏洞。`adversarial-review` 审查对象是描述性文档，找决策/假设/需求层面的漏洞。两者不重叠，可能会在同一项工作中先后使用（先用 `adversarial-review` 挑文档，改完提案再写代码，再用 `code-review` 挑代码）。
- **`grilling`**：交互式、逐条追问用户本人，适合"手头有个还没定型的想法，想被逼着想清楚"的场景。`adversarial-review` 是对已成文文档的一次性自动分析，产出结构化报告，不需要用户实时问答。SKILL.md 中会提示：如果想针对某条发现继续深挖讨论，可以转而使用 `grilling`。

## 后续扩展路径（不在本次实现范围）

先把规格类文档的审查跑扎实。以后如果需要覆盖更泛化的文本类型（博客草稿、决策记录等），再放宽"文档类型"的假设——五阶段流水线本身的结构不需要为此大改。
