# explain — 生成一页"看图 + 看例子"的代码业务逻辑讲解

## 这个 skill 是做什么的

用户指给你一段代码（位置或片段），你的任务不是把语法翻译一遍，而是讲清楚它在**业务上**做了什么、为什么这么做，并把讲解产出成**一个自包含的 HTML 页面**——写到文件、用浏览器打开。最有效的两个手段是：

1. **具体的走查例子** —— 选一组真实输入，顺着代码跑一遍，展示中间值如何变化，最后得到输出。这是让人"看懂"逻辑最快的方式。
2. **图表** —— 把控制流、调用关系、状态迁移、时间区间、规则组合可视化。一张对的图胜过三段文字。

页面用项目自带的模板渲染（专业现代、浅色为主可切深色、顶部导航 + 导出 PDF + 图可缩放、移动端与打印友好），适合内部技术文档和业务汇报。读者是开发者，他要的是"心智模型"，不是 API 文档。

## 核心原则：讲业务，不要复述语法

逐行翻译语法是低价值的。要把"机器语言"翻译成"业务语言"。

- ❌ 差："这里 `if (entryTime.isBefore(rentStartDate))` 判断进场时间是否早于租期开始时间，然后 `add` 一个对象到 list。"
- ✅ 好："如果车在**买长租之前就已经进场**，那么'进场 → 长租生效日'这段空档不在长租覆盖范围内，要单独按临停计费，所以这里把这段时间切出来排队等会儿算钱。"

把魔法值翻译成含义（`feePayFlag="T"` → 无需缴费；`orderStatus="1"` → 已支付），把变量名背后的业务概念说出来。

## 工作流程

1. **定位并读取目标**：解析 `file:line`、函数名、文件路径或粘贴的片段并读完。指向模糊时先用 Grep/Glob 找到最匹配的位置，必要时简短确认。
2. **顺着依赖追上下文（关键，适可而止）**：不要在真空里解释。为讲清**这段**代码通常要跳一跳看——它读写的**实体/字段**含义（金额、时间常用 String 存）、**被调用的服务/仓储/工具**到底干了啥（跟进去看，别只凭方法名猜）、**枚举/常量**背后的业务语义、以及**数据来源**。很多后端是 `Controller → Service → Repository/DAO → SQL/ORM 映射` 分层；当逻辑是"数据形态"的（查询/过滤/映射），追到真正的 SQL / 映射文件去看（MyBatis 项目尤其要看对应 mapper XML），比在 Service 层猜靠谱。**追到能把业务故事讲完整就停**，不要读穷整棵调用树。
3. **提炼业务意图**：动手前先用一两句话说清"这段代码为谁、解决什么现实问题"。需要推断的业务意图，标注为推测。
4. **选图**：根据逻辑"形状"选合适的图（见下表）。
5. **生成 HTML 解释页**：按 8 区块结构拼正文、用装配脚本出页面并打开（见"生成 HTML 解释页"）。
6. **在对话里给一段简短小结 + 页面路径**：让用户不切窗口也能知道大意，并能找到/保存这个页面。

## 选哪种图：按代码形态对症下药

| 代码形态 | 推荐图 | 为什么 |
| --- | --- | --- |
| 多分支 if/else、卫语句、嵌套条件 | **流程图** flowchart | 把嵌套条件摊平成一条条可走的路径 |
| 跨服务/跨层调用、消息、回调、开闸/播报等外部动作 | **时序图** sequence | 看清谁调谁、先后顺序、往返与副作用 |
| 实体带状态字段、生命周期（订单状态、审批流） | **状态图** state diagram | 状态如何迁移、什么条件触发迁移 |
| 枚举分支、规则表、参数组合（"每种情况怎么办"） | **决策表 / 表格** | 把所有 case 和处理列清楚，避免漏 |
| 基于时间窗/周期的计算（跨天计费、有效期分段） | **时间轴** timeline（手写内联 `<svg>` 涂色区间） | 把时间区间画出来，重叠/空档一目了然 |
| 金额累加 / 区间求差 / 公式（应付−已付=待付） | **公式流 / 小流程图**（数值用 `.val` 高亮） | 把算式画出来，比文字描述好懂 |
| 两种情况对比（A vs B，如 ROAD vs SEAL） | **并排双图** / 对照表 | 差异点并列，一眼看出区别 |
| 数据在层间转换（DTO→Entity→VO、SQL→对象） | **数据流图 / 映射表** | 字段怎么映射、在哪一层变形 |

一段代码可能需要不止一张图（先用流程图讲分支，再用时间轴讲其中一支的时间计算）。但**不要为画图而画图**——简单代码一张小图或一个例子就够。

## 生成 HTML 解释页

页面骨架在 `assets/template.html`：**顶部导航栏、深色切换、导出 PDF、Mermaid 缩放/重置、规则表搜索筛选、移动端与打印优化**都已内置，且 **Tailwind 与 Mermaid 已内联**（自包含、离线、**绝不引用任何在线 CDN**——目标环境常无外网或被墙，依赖 CDN 会导致样式/图坏掉甚至页面卡死）。你只负责写正文内容。

**动手前先读 `references/page-structure.md`** —— 它给出第 2–8 区块的逐块 HTML 模板、组件类、以及交互所需的 JS 钩子约定（可缩放图、规则表搜索筛选、手风琴等）。页面**必须严格按这 8 个部分依序组织**：

1. 顶部导航栏（模板提供，由 `--title` / `--path` 填充）
2. Hero + 关键指标卡片
3. 执行摘要（业务高层概述）
4. 核心业务流程（大 Mermaid 图 + 控制按钮 + 步骤列表）
5. 详细逻辑拆解（手风琴，每步含业务含义 + 技术说明）
6. 业务规则矩阵（可搜索 + 风险筛选的表格，风险用彩色徽章）
7. 风险与改进建议（卡片网格，按严重程度分类）
8. 附录（分析对象、相关文件、时间等元数据）

**装配流程**：把第 2–8 区块的 HTML 写到 `content.html`，再跑脚本（注入标题/路径，并内联 Tailwind + Mermaid，产出离线单文件）：

```
python "<本skill目录>/scripts/build_explainer.py" \
  --title "OrderService.updateOrderWhenPaySuccess 支付回调" \
  --path  "src/main/java/.../OrderService.java#updateOrderWhenPaySuccess" \
  --content content.html \
  --out "<临时目录>/explain-<对象名>-<时间戳>.html"
```

输出放系统临时目录（避免污染用户仓库），把路径告诉用户方便保存。打开：Windows `start "" "<file>"`；macOS `open "<file>"`；Linux `xdg-open "<file>"`。

**图表优先（核心要求）**：默认用图表表达，文字只做补充。每写一段先问"**这能不能画成图？**"——能就画（哪类逻辑画哪种图见上面《选哪种图》表）。别用大段文字或长 `<ul>` 描述流程/分支/关系，改成图、彩色卡片或手风琴。一个复杂方法通常该有**不止一张图**（核心流程图 + 状态图/时间轴/时序图常常都要）。颜色统一：流程图里某分支用什么色，对应卡片/徽章就用同色，读者一眼能对上。唯一红线：每张图都要真的帮助理解，别把两三行数据硬包成假图表凑数。

**图的画法**：标准图用 `<pre class="mermaid">`（`flowchart`/`sequenceDiagram`/`stateDiagram-v2`/`gantt`），中文标签**务必加引号** `A["订单=已支付"]`，否则 `()=+:` 会让解析失败；定制图（如收费/免费时间段涂色的时间轴）在 `<div class="diagram">` 里手写内联 `<svg>`。每张图都放进 `<div class="diagram">` 才会获得缩放/重置工具条。

**降级**：
- 若 `assets/tailwind.js` / `assets/mermaid.min.js` / 脚本不可用：仍尽量按 8 区块结构出页面，Mermaid 缺失时图改手写内联 `<svg>`。
- 若环境根本不适合生成/打开 HTML（无浏览器/无头，或用户明确要在对话里看）：退化为对话里直接 Markdown 讲解，图用 ```mermaid``` 代码块或 ASCII——内容与原则不变。

## 怎么走一个好例子

例子是这个 skill 的灵魂：

- **选有代表性的真实输入**：贴近真实场景的车牌、时间、金额、状态，而不是 `foo`/`123`。
- **缺少的外部规则要标注**：若计算依赖代码外的规则/费率（如单价存在数据库），要么查到真实值，要么**明确标注是为讲解假设的示例值**，别让读者以为是系统真实规则。
- **一步步展示状态变化**：用步骤条 `<ol class="steps">` 展示关键变量/字段在每一步的值（数值用 `<span class="val">` 高亮），让读者能跟着"心算"。
- **主路径 + 边界各一**：先走正常路径，再挑一个有意思的边界（空集合、跨界、首尾不对齐等）——边界往往才是代码复杂的原因。

例："长租车跨租期分段计费"可画一条时间轴，标出进场、长租生效、长租到期、出场四点，把"收费段/免费段"涂色，再用一组具体日期算出每段金额相加。

## 准确性与诚实（解释类任务的底线）

错的解释比不解释更糟。

- 每个结论都落在**真实代码**上，读到哪说到哪，不脑补不存在的分支或字段。
- 区分"代码写了什么"与"我推测的业务意图"——后者标注为推测。
- 例子里凡代码没定义的数值（费率、阈值），明确标注是示例假设值。
- 拿不准就说拿不准，并指出该看哪个文件/配置确认，而不是编一个圆满的故事。

## 把握深度与语言

- **深度匹配复杂度**：小工具函数 → 一个例子可能就够（页面可以很短，部分区块可精简）；上百行的分支大方法 → 值得流程图 + 时间轴 + 走查 + 完整 8 区块。也读用户诉求：他想"快速懂个大概"还是"彻底搞清每个分支"。
- **语言跟随用户**：用户中文就中文、英文就英文。注释/术语为中文的项目默认中文，并**保留代码标识符原样**（变量名、方法名、魔法值不翻译）。
